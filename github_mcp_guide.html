GitHub MCP Server

How Claude connects to GitHub, what tools it gets, and how everything works end-to-end.

Open Source Official GitHub 105+ Tools April 2025 Launch

What is MCP?

Model Context Protocol (MCP) is an open standard (created by Anthropic) that lets AI models like Claude talk to external tools and services through a unified interface. Think of it as a universal plugin system for AI — instead of custom integrations per tool, any MCP-compatible server can plug in.

GitHub's official MCP server translates Claude's tool calls into real GitHub API operations — creating issues, reading files, reviewing PRs, scanning for secrets — all through natural language.

ℹ️
Architecture in one line: Claude → MCP Protocol → GitHub MCP Server → GitHub REST/GraphQL API → your repos, issues, PRs, actions.

The GitHub MCP server is open source, written in Go, and built in collaboration with Anthropic. It launched in public preview in April 2025 and as of early 2026 exposes 105+ tools across 19 toolsets.

How Claude Connects to GitHub MCP

Claude connects through an MCP host configuration. There are two deployment modes:

☁️ Remote Server (Recommended)

GitHub hosts the MCP server. Claude connects to it over HTTPS. Zero local install needed.

https://api.githubcopilot.com/mcp/

🐳 Local Server (Docker / Binary)

Run the MCP server locally via Docker or a compiled binary. Communicate over stdio or local HTTP.

ghcr.io/github/github-mcp-server

Connection Flow (step by step)

1

Claude reads your MCP config

On startup, Claude Desktop reads claude_desktop_config.json (or Claude Code reads .mcp.json / project settings). It finds the GitHub MCP server entry under "mcpServers".

2

MCP handshake & tool discovery

Claude connects to the server (via stdio for local, HTTPS for remote) and calls tools/list. The GitHub MCP server responds with the full schema of every enabled tool — name, description, input parameters.

3

Tool schemas loaded into context

Claude now "knows" about all available GitHub tools. This is why toolset filtering matters — loading all 105+ tools consumes significant context window space (~150K tokens total if unconstrained).

4

User asks a natural language question

For example: "Show me all open PRs labeled bug in the api-server repo". Claude decides which tool to call and with what parameters.

5

Claude issues a structured tool call

Claude emits a structured call like list_pull_requests(owner="acme", repo="api-server", labels=["bug"], state="open") over the MCP protocol.

6

GitHub MCP server calls the GitHub API

The server translates the tool call into a GitHub REST or GraphQL API request, using your token for authentication. It validates inputs against the schema before making the request.

7

Result returned to Claude

GitHub API responds → MCP server structures the result → Claude receives it as tool output → Claude synthesizes a natural-language answer for you.

Authentication

The GitHub MCP server supports three authentication methods. Your token is never sent to Claude — it lives in your local config or environment and is used only by the MCP server itself when calling the GitHub API.

🔑 Personal Access Token (Classic) COMMON

Set GITHUB_PERSONAL_ACCESS_TOKEN env var. Good for personal use. Scoped permissions (repo, read:org, etc.). The server auto-filters tools based on your granted scopes.

🔐 Fine-Grained PAT RECOMMENDED

Modern token format with per-repo and per-permission granularity. GitHub enforces permissions at API level. More secure, especially for production use.

☁️ OAuth (Remote server) EASIEST

One-click OAuth flow via GitHub App. Works with VS Code 1.101+ and compatible hosts. Interactive scope up-grading available inline.

🏢 GitHub App Token ENTERPRISE

Server-to-server auth for GitHub Enterprise Server. Supports installation tokens. Under active development as of early 2026.

💡
OAuth scope filtering (Jan 2026): With Classic PATs, the server automatically hides tools your token doesn't have permission to use. You won't see a create_issue tool if your PAT lacks repo scope.

Step-by-Step Setup

Option A — Remote Server (Easiest)

1

Generate a GitHub PAT

Go to GitHub → Settings → Developer settings → Personal access tokens. Create a fine-grained token with the repo permissions you need (read contents, write issues, etc.).

2

Add to Claude Code via CLI

claude mcp add-json github '{
  "type": "http",
  "url": "https://api.githubcopilot.com/mcp",
  "headers": {
    "Authorization": "Bearer YOUR_GITHUB_PAT"
  }
}'
bash
3

Verify connection

Start Claude Code and type: "List my GitHub repositories". Claude will call the MCP server and return results if connected correctly.

Option B — Local Docker Server

1

Pull the Docker image

docker pull ghcr.io/github/github-mcp-server
bash
2

Edit claude_desktop_config.json

macOS: ~/Library/Application Support/Claude/claude_desktop_config.json
Windows: %APPDATA%\Claude\claude_desktop_config.json

{
  "mcpServers": {
    "github": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-e", "GITHUB_PERSONAL_ACCESS_TOKEN",
        "ghcr.io/github/github-mcp-server"
      ],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_your_token_here"
      }
    }
  }
}
json
3

Restart Claude Desktop

Claude Desktop will spawn the Docker container on startup and perform the MCP handshake automatically. A small plug icon appears in the UI when MCP servers are connected.

Toolsets & Feature Flags

Toolsets are named groups of related tools. You select which toolsets to enable rather than individual tools (unless you use the X-MCP-Tools header for fine-grained control).

Default toolsets (auto-enabled)

repos DEFAULT
Browse files, search code, view commits
issues DEFAULT
Create, read, update, list issues
pull_requests DEFAULT
PR lifecycle, reviews, merges
users DEFAULT
User profiles, org membership
context DEFAULT
Core MCP context tools

Optional toolsets

Workflow runs, logs, CI/CD
code_security
CodeQL alerts, Dependabot

Leaked credential detection

GitHub Projects management

Team discussions
notifications
Notification management
gists
Gist create/read/update

Experimental features

Copilot-specific tools

Remote server only — search GitHub docs

Enabling toolsets

# Environment variable (Docker)
GITHUB_TOOLSETS=default,actions,code_security

# CLI flag (local binary)
github-mcp-server --toolsets repos,issues,pull_requests,actions

# HTTP header (remote server)
X-MCP-Toolsets: default,code_security

# Keywords
default    # = repos + issues + pull_requests + users + context
all        # = every available toolset
bash

Security flags

# Read-only mode — disables ALL write operations
GITHUB_READ_ONLY=1                    # Docker env var
github-mcp-server --read-only         # CLI flag

# Dynamic toolset discovery (reduces initial context load)
GITHUB_DYNAMIC_TOOLSETS=1            # Starts with 3 discovery tools only

# Insiders (experimental features, Jan 2026+)
GITHUB_INSIDERS=1
X-GitHub-Insiders: true               # HTTP header for remote
bash

Where Tools Are Defined

Tools are defined server-side in the open-source GitHub MCP server repository. Claude never defines them — it only discovers them via the MCP handshake at runtime.

github.com/github/github-mcp-server
Root of the open-source Go server. All tool definitions live here.
pkg/github/*.go
Per-category Go files. Each file defines tools for a domain (e.g. issues.go, pull_requests.go, repositories.go). Tool name, description, input schema (JSON Schema), and handler function are all co-located.
pkg/toolsets/toolsets.go
Maps toolset names (e.g. "repos", "actions") to their constituent tools. This is where the grouping logic lives.
cmd/github-mcp-server/main.go
Entry point. Parses env vars (GITHUB_TOOLSETS, GITHUB_READ_ONLY, etc.), initialises the MCP server, registers toolsets, and starts listening (stdio or HTTP).
pkg/github/server.go
Server instructions — acts like a system prompt for the MCP server, guiding Claude on tool interdependencies, pagination patterns, and multi-tool workflows.
~/.claude/claude_desktop_config.json
Your local config. Tells Claude where the server is and what credentials to use — but does not define any tools.
ℹ️
Tool schema format: Each tool is a JSON Schema object with name, description, and inputSchema (properties, types, required fields). Claude uses the description to decide when to call a tool and the inputSchema to know what parameters to pass.

Example: how a tool is defined in Go (simplified)

// pkg/github/issues.go
mcp.NewTool("create_issue",
  mcp.WithDescription("Create a new GitHub issue in a repository"),
  mcp.WithString("owner",  mcp.Required(), mcp.Description("Repository owner")),
  mcp.WithString("repo",   mcp.Required(), mcp.Description("Repository name")),
  mcp.WithString("title",  mcp.Required(), mcp.Description("Issue title")),
  mcp.WithString("body",   mcp.Description("Issue body (Markdown)")),
  mcp.WithArray("labels",  mcp.Description("Labels to apply")),
)
go

Advantages of GitHub MCP

🗣️ Natural Language First

No command memorization. "Find all open PRs from last week that touch the auth module" maps directly to a search tool call with the right parameters.

🔗 Multi-Step Workflows

Claude can chain multiple tool calls. "Review the latest PR, check if tests passed, and post a summary comment" = 3 tools in sequence, all in one request.

🔐 Per-User Governance

OAuth scopes are per-user. Each person's token controls exactly what they can do. Fine-grained PATs add per-repo permission control on top.

📋 Structured Input Validation

Every tool call is validated against a JSON Schema before hitting the GitHub API. No malformed requests, better error messages, safer automation.

🔒 Read-Only Mode

GITHUB_READ_ONLY=1 disables all write tools at the server level. Perfect for security-sensitive environments where you want AI with eyes but no hands.

🏗️ Ecosystem Composable

MCP works alongside Slack MCP, Jira MCP, and others. Claude can pull a GitHub PR, check the linked Jira ticket, and post a Slack update — all in one conversation.

📡 Server Instructions (Oct 2025)

The server ships with built-in guidance for Claude — teaching it the right tool sequencing for workflows like "review a PR" or "investigate a failing Action."

🔍 Dynamic Toolsets (2026)

Start with only 3 discovery tools in context. Load additional toolsets on demand. Keeps context lean and fast for simple tasks.

MCP vs gh CLI vs GitHub REST API

DimensionGitHub MCPgh CLIGitHub REST API
Setup complexity Medium (config file) Low (brew install gh) High (manual HTTP calls)
Natural language Yes — full NL interface Partial (you write commands) No
Context window cost ~150K tokens (all tools) ~2K tokens (CLI in training) Depends on usage
Multi-step automation Built-in, guided by server instructions Possible via scripting Possible but manual
Permission governance OAuth, scope filtering, read-only mode PAT scopes only PAT or OAuth token
Input validation JSON Schema at protocol level CLI flag validation Manual
Speed / latency Slower (remote server) Fast (local binary) Fastest (direct)
Cross-tool composability Works alongside Slack, Jira, etc. GitHub only GitHub only
Audit trail Structured tool calls, OAuth per-user Shell history API logs
Best for Multi-system AI workflows, governance-heavy teams Quick scripted GitHub tasks Custom integrations
⚠️
Context cost reality check: Loading all MCP tools (~150K tokens) vs using gh CLI in bash (~2K tokens) is a 98.7% difference. For simple single-repo workflows, gh CLI called from Claude's bash tool is often more efficient. MCP wins when you need cross-system composability or strong governance.

Tool-specific control (Jan 2026)

# Only allow these 3 specific tools (via HTTP header)
X-MCP-Tools: get_file_contents,search_code,create_issue

# Combine tools + toolsets
X-MCP-Toolsets: default
X-MCP-Tools: list_code_scanning_alerts
http

Dynamic toolset discovery mode

# Only 3 discovery tools loaded initially
GITHUB_DYNAMIC_TOOLSETS=1

# Claude then calls these to discover and enable more:
list_available_toolsets()    # → shows what's available
enable_toolset("actions")    # → activates workflow tools
get_toolset_tools("actions") # → lists enabled tools
